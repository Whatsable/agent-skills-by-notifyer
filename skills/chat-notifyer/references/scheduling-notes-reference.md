# Scheduling & Notes Reference — chat-notifyer

## Scheduled Messages

### Overview

Scheduled messages allow you to queue WhatsApp messages to be sent at a future time. All three message types (text, template, attachment) support scheduling.

Scheduled messages are stored in Xano's `chat_schedule` table and executed automatically by a background job.

---

### List Scheduled Messages

```
GET /api:bVXsw_FD/web/chat_schedule
```

**Authentication:** `Authorization: <token>` (raw JWT)  
**CORS:** `Origin: https://chat.notifyer-systems.com`

**Response:** Array of scheduled message objects.

**Scheduled Message Fields:**

| Field                    | Type     | Description |
|-------------------------|----------|-------------|
| id                      | integer  | Schedule record ID — use for deletion |
| phone_number            | integer  | Recipient phone number |
| message_type            | text     | "text" \| "template" \| "attachment" |
| text                    | text     | Content (text messages only) |
| template                | text     | Template ID (template messages only) |
| variables               | json     | Template variables (template messages only) |
| url                     | text     | Media URL (attachment messages only) |
| mime_type               | text     | MIME type (attachment messages only) |
| caption                 | text     | Caption (attachment messages only) |
| scheduled_time          | integer  | Unix ms timestamp when to fire |
| status                  | text     | "pending" \| "sent" \| "failed" |
| created_at              | integer  | When the schedule record was created |

---

### Delete Scheduled Message

```
DELETE /api:bVXsw_FD/web/chat_schedule/:id
```

**Authentication:** `Authorization: <token>` (raw JWT)  
**CORS:** `Origin: https://chat.notifyer-systems.com`

**Path parameter:** `id` — integer schedule record ID

**Response:** HTTP 200 with empty body on success (script synthesises `{ deleted: true, id: <n> }`).

**Already-fired messages:** If the scheduled time has passed and the message was sent, the record may have already been cleared from the table. A 404 response indicates the record is gone.

---

### Scheduling Behaviour by Message Type

| Script             | immediate send      | scheduled send |
|-------------------|---------------------|----------------|
| send-text.js      | Omit `scheduled_time` | Include `scheduled_time: <unix_ms>` |
| send-template.js  | `scheduled_time: 0` | `scheduled_time: <unix_ms>` |
| send-attachment.js | `scheduled_time: 0` | `scheduled_time: <unix_ms>` |

**Xano implementation note:** `send-text.js` uses `is_empty` check on the input field, so omitting the key entirely triggers immediate send. `send-template.js` and `send-attachment.js` use `!= 0` check, so must always include the field.

---

### Date Format for Scripts

Scripts accept: `"DD/MM/YYYY HH:mm"` e.g. `"25/01/2025 14:00"`

Conversion: scripts parse this locally and compute Unix ms timestamp. The timezone is the system timezone of the machine running the script. If running in a different timezone, adjust accordingly.

## Notes

### Two Note Types

Notifyer tracks two separate note fields on each recipient:

| Field     | Description | Editable via Script |
|-----------|-------------|---------------------|
| note      | Manual note | Yes — via add-note.js |
| note_auto | AI-generated note (conversation summary) | No (read-only from API) |

---

### Manual Notes (`note`)

Free-text field visible in the conversation sidebar of chat.notifyer-systems.com.

**Use cases:**
- "VIP customer — apply 20% discount automatically"
- "Prefers communication in Spanish"
- "Do not contact on weekends"
- "Follow up about renewal on 15 Feb"

**Operations:**
- Read: `get-notes.js`
- Set/replace: `add-note.js --note "new note text"`
- Append: `add-note.js --append "additional info"`
- Clear: `add-note.js --note ""`

**Endpoint:**
```
PATCH /api:bVXsw_FD/web/recipient/:id
Body: { "note": "new note text" }
```

**CORS:** `Origin: https://chat.notifyer-systems.com`

---

### AI-Generated Notes (`note_auto`)

Auto-generated by Notifyer's AI during conversation processing. Contains a summary of the conversation context (e.g., "Customer asked about order status for #12345, resolved on Jan 25").

**Characteristics:**
- Written automatically by Xano AI logic — not by agents
- Cannot be set or cleared via API (read-only from the API surface)
- Updated continuously as conversations progress
- Visible in conversation sidebar alongside manual note

**Read only:** `get-notes.js` returns both `note` and `note_auto`.

---

### Fetch Strategy for Notes

Notes are stored on the recipient record. To read or update them:

1. `GET /web/recipient?search=<phone>` → find recipient → read `note` and `note_auto`
2. `PATCH /web/recipient/:id` with `{ note: "..." }` → update manual note

No dedicated notes endpoint exists — notes are part of the recipient object.

---

## Common Errors

| Error | Cause | Fix |
|-------|-------|-----|
| HTTP 404 on delete | Scheduled message already fired or not found | Check `list-scheduled.js` first |
| Scheduled message not sent | Xano background job may have failed | Check status field via `list-scheduled.js` |
| Note not persisting | PATCH returned error | Check auth and CORS; verify recipient ID |
| note_auto is null | AI hasn't processed conversation yet | Normal — AI note populates over time |
